## Overview

Personal portfolio site for Molly Chen. Built with Vite + React, deployed on GitHub Pages. The `reactMigration` branch holds the current React codebase; `master` holds the previous static HTML version.

## Running locally

```bash
npm run dev
# then open http://localhost:5173
```

## Building for deployment

```bash
npm run build   # outputs to docs/
```

Configure GitHub Pages to serve from the `docs/` folder on the branch you want to deploy.

## Architecture

### Tech stack

- **Vite** — build tool, dev server
- **React 18** — UI
- **React Router v6** (HashRouter) — client-side routing; HashRouter is required for GitHub Pages since there's no server-side routing

### Pages

| File | Route | Purpose |
|---|---|---|
| `src/pages/HomePage.jsx` | `/#/` | About, Experiences, Skills |
| `src/pages/PortfolioPage.jsx` | `/#/portfolio?tab=<tabId>` | Portfolio with Consumer Apps / Business Integrations tabs |
| `src/pages/ProjectPage.jsx` | `/#/project/:id` | Individual project detail page |

### Shared components

| File | Purpose |
|---|---|
| `src/components/Navbar.jsx` | Nav bar — shared across all pages |
| `src/components/Header.jsx` | Header with headshot and name — shared across all pages |
| `src/components/Footer.jsx` | Footer with social links — shared across all pages |
| `src/App.jsx` | Router + `BackgroundSlideshow` component (two-layer crossfade) |

### Data flow for portfolio content

`public/data/portfolio.json` is the single source of truth for all project cards.

1. **`src/lib/portfolioDb.js`** — exports `loadPortfolioPayload()`. Fetches `portfolio.json`, writes it to IndexedDB as a snapshot, returns `{ data, source }`. Falls back to the cached snapshot on network failure.
2. **`src/lib/slugify.js`** — shared `slugify()` used by both `PortfolioPage` (card links) and `ProjectPage` (matching `?id=` to a card). Slugify: lowercase, em/en-dashes → `-`, strip non-alphanumeric, collapse hyphens.
3. **`PortfolioPage`** — calls `loadPortfolioPayload()`, renders sections and cards as React components.
4. **`ProjectPage`** — calls `loadPortfolioPayload()`, matches `params.id` via `slugify()`, renders the detail view.

### Routing

- **Tab switching on PortfolioPage** — driven by `?tab=` query param (`useSearchParams`). Clicking a tab in the nav dropdown navigates to `/#/portfolio?tab=<tabId>`, which re-renders with the correct section visible. The active section gets the CSS class `consumer-apps`.
- **Project detail pages** — `/#/project/<slug>`, where slug is generated by `slugify(card.title)`. No separate routing config needed; slug is derived from the title at render time.
- **In-page anchor links** (Experiences, Skills on HomePage) — when navigating from another page, `Navbar` passes `state={{ scrollTo: '<id>' }}` via React Router. `HomePage` reads `location.state` in a `useEffect` and calls `scrollIntoView`.

### CSS scoping

`public/css/custom.css` uses `body.portfolio-page` to scope portfolio-specific styles. `HomePage` clears this class; `PortfolioPage` and `ProjectPage` set it (and `project-page` respectively) via `useEffect`, cleaning up on unmount.

Notable quirk: lines ~435–441 in `custom.css` have `.about, .languages, .contact { background: #eafff0 ... }` — this was intended to style `.status-pill` descendants but accidentally targets the section elements themselves. It is overridden by more-specific rules and should not be "fixed" without checking cascade effects.

## Adding or editing project cards

Edit `public/data/portfolio.json`. Each card under a section's `cards` array supports:

```json
{
  "title": "...",         // required — used to generate the project page slug
  "status": { "label": "Category Name" },
  "problem": "HTML string",
  "metrics": "HTML string",
  "solution": "HTML string",
  "previewUrl": "https://..."   // optional — renders an iframe preview on the card and detail page
}
```

Each section also supports `"titleIcon"` (Font Awesome class) and `"titleIconStyle"` (CSS string, e.g. `"color: #3366cc"`).
